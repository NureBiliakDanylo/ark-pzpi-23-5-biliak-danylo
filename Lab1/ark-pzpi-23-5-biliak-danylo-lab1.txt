Після цього сформувалася концепція системи – набір ключових ідей, що визначають її функціональність, структуру та відмінні риси.
На наступному етапі відбувалася деталізація вимог і узгодження основних компонентів системи. У документі було описано загальну архітектуру, межі проєкту, очікувані результати та ключові критерії успіху. Для кожного модуля визначено головні завдання, способи взаємодії з іншими частинами системи та рівень пріоритетності.
Завершальним етапом стало узагальнення отриманих результатів у єдину структуру документа. В Vision & Scope були зведені всі ключові аспекти: від концепції та цілей до технічних вимог і перспектив розвитку системи. ДОДАТОК Б
1.1 Аналіз проблеми
Більшість сервісів прогнозу погоди надають усереднені дані для великих міст або регіонів. Через це користувачі не отримують точного прогнозу для конкретної місцевості – наприклад, для села, передмістя або ферми.
        Крім того, дані оновлюються з затримкою, і немає можливості бачити фактичні вимірювання з реальних локальних сенсорів.
Система покликана вирішити цю проблему за допомогою простих IoT-пристроїв, що фіксують температуру, вологість і тиск, та надсилають ці дані на сервер, який обробляє їх і відображає у зручному веб- або мобільному інтерфейсі.


1.2 Аналіз аналогів
Існуючі сервіси: OpenWeather, AccuWeather, Gismeteo – дають загальні прогнози, але не враховують локальні умови.
        Деякі комерційні рішення з IoT-станціями дорогі й закриті для користувачів.
Недоліки:
   * Немає точних локальних даних.
   * Відсутня інтеграція власних сенсорів.
   * Складність у використанні або висока ціна.

Нова система дозволить зібрати дані з власних недорогих IoT-пристроїв і показати прогноз на основі реальних вимірювань користувача або його громади.


2.1 Опис призначення системи
Система призначена для збору й відображення локальних погодних даних у зручній формі.
        Користувач може бачити поточну температуру, вологість, атмосферний тиск і простий прогноз на кілька годин або днів.
Цінність:
      * Отримання точного прогнозу для конкретного місця.
      * Можливість підключити власний сенсор.
      * Простий доступ із браузера або мобільного пристрою.

2.2 Опис бізнес-логіки
         1. IoT-пристрій вимірює температуру, вологість і тиск.
         2. Дані надсилаються на сервер (HTTP-запит).
         3. Сервер зберігає показники в базі даних і виконує базовий розрахунок прогнозу (середнє, тренди).
         4. Користувач через веб або мобільний інтерфейс може:
         * Переглянути поточні показники;
         * Подивитись графік за останню добу;
         * Переглянути короткий прогноз;
         * Зареєструвати власний сенсор.


3.1 Опис функціональності серверної частини та IoT-клієнта
Серверна частина:
         * Прийом даних від IoT-пристроїв.
         * Збереження у базі даних.
         * Простий аналіз (середні значення, мін/макс).
         * API для отримання даних веб- і мобільним клієнтами.
IoT-клієнт:
         * Вимірює температуру, вологість, тиск.
         * Надсилає дані кожні N хвилин на сервер через HTTP.
         * Може бути побудований на основі ESP8266 або Raspberry Pi.


3.2 Опис функціональності веб-клієнта та мобільного клієнта
Веб-клієнт:
         * Відображення поточних даних.
         * Графік зміни температури, вологості та тиску.
         * Простий інтерфейс для перегляду прогнозу.
Мобільний клієнт:
         * Відображення тих самих даних у спрощеному вигляді.
         * Адаптивний дизайн для телефонів.
         * Отримання оновлень через API.
Взаємодія із сервером:
         * Через REST API (JSON-запити та відповіді).


4.3 Опис програмних засобів та інструментів
Проект складається з кількох основних частин. Серверна частина реалізована на Node.js з використанням фреймворку Express. Для збереження даних використовується база даних SQLite або PostgreSQL, а обмін інформацією між компонентами відбувається через REST API у форматі JSON. IoT-пристрій побудований на базі плати ардуіно та програмується мовою C++ у середовищі Arduino IDE. Він передає дані на сервер за допомогою HTTP POST запитів. Веб-клієнт створюється з використанням HTML, CSS і JavaScript, можливо, із застосуванням фреймворку React або через прості запити Fetch API. Для мобільного клієнта використовується Flutter, що забезпечує кросплатформність застосунку. Серед інструментів розробки передбачено GitHub для керування версіями коду та VS Code або Arduino IDE для роботи з серверною частиною та мікроконтролером відповідно.
